## 2. Опис виконання практичних завдань

### Завдання 1-2: Зміна State "на льоту" за допомогою React DevTools
Під час розробки списку коктейлів я тестував функціонал додавання напою до улюбленого. Замість того, щоб щоразу проходити весь шлях в інтерфейсі (що могло вимагати підключення до API та реальних кліків), я знайшов компонент `CocktailCard` у вкладці **Components** (React DevTools).  
У правій панелі у секції `hooks` / `State` я знайшов змінну `isFavorite: false` і вручну змінив її значення на `true`. Завдяки цьому я миттєво побачив, як іконка на UI змінилася на зафарбоване сердечко без перезавантаження сторінки.

### Завдання 3: Робота з нескінченним циклом (Too many re-renders)
Я розробляв кнопку для того, щоб ставити лайк конкретному коктейлю. Однак, при спробі відобразити макет, сторінка крашнулась, а в консолі з'явилась помилка: `Error: Too many re-renders. React limits the number of renders to prevent an infinite loop.`

Помилка виникла через неправильний виклик функції оновлення стану в обробнику події, ось шматок мого неправильного коду:

```jsx
const CocktailLikeButton = ({ initialLikes }) => {
  const [likes, setLikes] = useState(initialLikes);

  // ПОМИЛКА: setLikes викликається одразу при рендері
  return (
    <button onClick={setLikes(likes + 1)}>
      Лайків: {likes}
    </button>
  );
};
```
**Чому сталась помилка (стек викликів)?**
Функція `setLikes` викликається *безпосередньо під час побудови компонента*, адже я передав її виклик із дужками `()`. Оскільки виклик цієї функції змінює стейт, це ініціює новий рендер. Під час цього нового рендеру знов одразу викликається `setLikes`, що провокує ще один рендер, роблячи цикл викликів нескінченним. 
Щоб це виправити, я обгорнув зміну стану у стрілкову функцію: `onClick={() => setLikes(likes + 1)}`.

### Завдання 4-5: Використання debugger
Для перевірки структури даних, які приходять з API для списку коктейлів, я додав ключове слово `debugger;` безпосередньо у хук `useEffect` перед тим, як записати дані у стейт:

```jsx
useEffect(() => {
  const fetchCocktails = async () => {
    try {
      const response = await fetch('https://www.thecocktaildb.com/api/json/v1/1/search.php?s=margarita');
      const data = await response.json();
      
      debugger; // Зупиняємо виконання для перевірки формату data
      
      setCocktails(data.drinks || []);
    } catch (error) {
      console.error("Помилка завантаження", error);
    }
  };
  
  fetchCocktails();
}, []);
```
Маючи відкриту панель розробника через (F12), я оновив сторінку. Браузер автоматично поставив процес виконання на паузу на рядку з `debugger;`. У правій панелі (вкладка Sources) у секції **Scope / Local** я розгорнув об'єкт `data` і побачив, що коктейлі дійсно приходять у масиві під ключем `drinks`. Далі я натискав **F10 (Step Over)**, щоб покроково перейти до наступного рядка коду та перевірити безпечність виклику `setCocktails`.

## 3. Відповіді на контрольні запитання

**1. В чому полягає критична перевага використання React Developer Tools (Components) порівняно зі звичайним інспектуванням DOM-елементів (Elements) при пошуку помилок у React-додатку?**
Вкладка *Elements* показує лише фінальний відрендерений HTML, який бачить браузер. Натомість *React DevTools (Components)* відображає архітектуру застосунку — дерево віртуального DOM. Перевага в тому, що розробник має доступ до внутрішнього стану (`State`), властивостей (`Props`) та контексту конкретного компонента. Можна змінювати ці дані "на льоту" і тестувати поведінку UI без того, щоб розбиратися, як дані переведені в DOM.

**2. Як відловити ситуацію, коли React виконує неоптимальний рендеринг списків через неправильне використання атрибута key?**
По-перше, потрібно звертати увагу на попередження у браузерній консолі `Warning: Each child in a list should have a unique "key" prop`. По-друге, в React DevTools слід натиснути іконку налаштувань та увімкнути *"Highlight updates when components render"*. Якщо при зміні одного елемента (наприклад, додаванні коктейлю) контур підсвічує весь список об'єктів — це явна ознака того, що React не зміг зіставити елементи і перемальовує їх повністю (часто стається, якщо у якості ключа помилково використано індекс масиву `index`, а не унікальний `id`).

**3. Поясніть алгоритм дій розробника при встановленні та використанні Breakpoint у вкладці Sources браузера Chrome. Що відбувається з виконанням програми в цей момент?**
Алгоритм:
1. Відкрити інструменти розробника (F12) і перейти на вкладку **Sources**.
2. У дереві файлів (зліва) знайти потрібний JS файл (інколи захований у папках `webpack://` або `localhost`).
3. Вибрати рядок коду і клікнути на його номер, щоб встановити маркер (Breakpoint).
При взаємодії зі сторінкою, коли виконання скрипта досягає цього рядка, відбувається **Execution paused** (пауза виконання). Браузер заморожує інтерфейс, а розробник отримує змогу переглянути стек виклику функцій (Call Stack), подивитись поточні значення усіх локальних і глобальних змінних та продовжити виконання покроково.

**4. Яке ключове слово (резервоване в JavaScript) можна написати безпосередньо у вихідному коді компонента, щоб змусити браузер автоматично зупинити виконання при відкритій панелі розробника?**
Ключове слово — `debugger;` (його прийнято додавати на потрібну строчку для тих самих цілей, що й Breakpoints).

**5. Що означають дії "Step Over", "Step Into" та "Step Out" у панелі управління дебагером браузера (Sources)?**
- **Step Over (F10):** Виконує поточний рядок коду і переходить до наступного. Якщо на цьому рядку є виклик зовнішньої функції — він виконує її повністю у фоні, не заходячи всередину неї, видаючи одразу результат.
- **Step Into (F11):** Змушує зайти всередину функції, яка викликається на даному рядку. Це використовують для детальної перевірки чужого або складного коду покроково в глибину.
- **Step Out (Shift + F11):** Миттєво довиконує поточну функцію до кінця без більше пауз у ній і "виходить" з неї на рівень вище — до того місця (наступного рядка), звідки цю функцію було початково викликано.
