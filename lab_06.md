## 1. Кастомний хук `useFetch`
**Файл:** `src/hooks/useFetch.js`

```javascript
import { useState, useEffect } from 'react';

/**
 * Кастомний хук для виконання GET-запитів.
 * @param {string} url - URL-адреса для запиту
 * @returns {{ data: any, isLoading: boolean, error: string | null }}
 */
export const useFetch = (url) => {
  const [data, setData] = useState(null);
  const [isLoading, setIsLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    // Створюємо AbortController для можливості відміни запиту
    const controller = new AbortController();
    const { signal } = controller;

    const fetchData = async () => {
      setIsLoading(true);
      setError(null);
      
      try {
        const response = await fetch(url, { signal });
        
        if (!response.ok) {
          throw new Error(`HTTP Error: ${response.status} ${response.statusText}`);
        }
        
        const result = await response.json();
        setData(result);
      } catch (err) {
        // Ігноруємо помилку, якщо вона викликана відміною запиту при розмонтуванні
        if (err.name !== 'AbortError') {
          setError(err.message || 'Щось пішло не так при завантаженні даних');
        }
      } finally {
        setIsLoading(false);
      }
    };

    if (url) fetchData();

    // Cleanup function: відміняє поточний запит, якщо компонент 
    // буде розмонтовано або зміниться залежність url
    return () => {
      controller.abort();
    };
  }, [url]);

  return { data, isLoading, error };
};
```

## 2. Компонент стрічки новин `NewsFeed`
**Файл:** `src/components/NewsFeed/NewsFeed.jsx`

```javascript
import React from 'react';
import { useFetch } from '../../hooks/useFetch';

// Гіпотетичний компонент для відображення картки одного поста
const PostCard = ({ title, body }) => (
  <article className="post-card">
    <h3>{title}</h3>
    <p>{body}</p>
  </article>
);

export const NewsFeed = () => {
  // Використовуємо кастомний хук
  const { data: posts, isLoading, error } = useFetch('https://jsonplaceholder.typicode.com/posts');

  // Тріада станів
  if (isLoading) {
    return <div className="loader">Завантаження новин...</div>; // Стан завантаження
  }

  if (error) {
    return <div className="error-message">Сталася помилка: {error}</div>; // Стан помилки
  }

  // Стан успіху (відображення даних)
  return (
    <div className="news-feed">
      <h2>Останні новини</h2>
      <div className="posts-list">
        {/* Рендеримо перші 10 постів */}
        {posts && posts.slice(0, 10).map((post) => (
          <PostCard key={post.id} title={post.title} body={post.body} />
        ))}
      </div>
    </div>
  );
};
```

## 3. Відповіді на контрольні запитання

**1. Поясніть призначення об’єкта AbortController у нашому хуку useFetch. Яку загрозу для React-додатку він усуває?**  
`AbortController` дозволяє скасувати мережевий запит посеред його виконання (шляхом передачі параметра `signal` у функцію `fetch`). У React цей механізм використовується всередині функції очищення (cleanup function) хука `useEffect`. Він усуває загрозу витоку пам'яті (memory leak) та помилки оновлення стану в *розмонтованому* (unmounted) компоненті. Це стається, коли запит ще в процесі, а користувач вже перейшов на іншу сторінку — після завершення запиту відбулася б спроба змінити стан неіснуючого компонента.

**2. Що таке патерн “Тріада станів” (loading, error, data) при роботі з мережею і чому він є обов’язковим для якісного UX?**  
Цей патерн передбачає обов'язкову обробку трьох можливих станів асинхронної операції:
- **Loading (в процесі):** показуємо лоадер/спінер. Це дає користувачеві зворотний зв’язок, що система працює, а не "зависла".
- **Error (помилка):** показуємо текст помилки. Пояснює користувачу, що пішло не так (наприклад, пропав інтернет) і пропонує щось зробити.
- **Data (успіх):** показуємо власне контент.
Без цієї тріади інтерфейс стає непередбачуваним і некомфортним, оскільки користувач не розуміє, на якому етапі знаходиться виконання його дії.

**3. Чому функція виконання мережевого запиту (fetch або axios.get) розміщується всередині хука useEffect, а не прямо в тілі функціонального компонента?**  
Тому що функціональний компонент викликається (рендериться) багато разів: щоразу при зміні його пропсів чи стану. Якщо розмістити `fetch` прямо в тілі, він буде виконуватися на кожен рендер. Оскільки запит потім зазвичай змінює стан (`setData`), це викличе новий рендер, який зробить новий запит, утворюючи **нескінченний цикл** (infinite loop). Хук `useEffect` дозволяє ізолювати побічний ефект (network request) і виконати його контрольовано — одноразово при монтуванні, або лише при зміні конкретних залежностей.

**4. У чому полягають головні переваги використання бібліотеки axios у порівнянні з нативним fetch?**  
- **Автоматична трансформація JSON:** не потрібно вручну викликати `response.json()`.
- **Зручніша обробка помилок:** Axios автоматично переводить запити з кодами статусу за межами "2xx" (наприклад, `404` або `500`) у блок `catch`. Native Fetch цього не робить і вважає запит "успішним", поки сервак дав хоч якусь (навіть помилкову) відповідь, змушуючи розробника писати перевірки на кшталт `if (!response.ok)`.
- **Інтерсептори (Interceptors):** зручний спосіб глобально перехопити всі запити для додавання, наприклад, JWT токенів або обробки помилок стандартизовано.
- **Широкий функціонал з коробки:** відстеження прогресу завантаження (upload progress), одночасні запити (axios.all), автоматична обробка таймаутів.

**5. Опишіть потенційні ризики багу “Стан гонитви” (Race Condition), якщо користувач буде дуже швидко перемикати сторінки з різними URL-параметрами, за якими робляться запити.**  
"Стан гонитви" (Race Condition) виникає, коли результати серії асинхронних запитів приходять у непередбачуваному порядку (старіший запит може відповідати довше, ніж новіший). Якщо користувач швидко перемкнеться з "Категорії 1" на "Категорію 2", браузер надішле два запити по черзі. Якщо відповідь для "Категорії 2" прийде миттєво, а для "Категорії 1" з затримкою, то інтерфейс спочатку відобразить правильні дані, а за мить "затреться" пізнішою відповіддю від старого запиту — на сторінці "Категорія 2" з'являться товари з "Категорії 1". Відміна запитів через `AbortController` або використання механізму ігнорування (cleanup boolean flags) вирішує цю проблему.
